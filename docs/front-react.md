# 프로젝트 아키텍처 설계서 (SSOT)

> 이 문서는 Claude Code 구현의 단일 기준점(Single Source of Truth)이다.
> 구현 중 이 문서와 충돌하는 결정을 내리지 말 것. 변경이 필요하면 먼저 이 문서를 갱신한다.

## 스택

- 런타임/번들러/패키지매니저/테스트러너: **Bun**
- 언어: **TypeScript**
- UI: **React** (웹 전용)
- React Native: **범위 밖** (이 프로젝트에서 고려하지 않음)

## 한 줄 요약

페이지별 SSR HTML + 이중 React 루트(영속 셸 / 교체 콘텐츠) + 1회용 화면키 캐시로
데이터 보간. "속도(빠른 첫 화면 + 빠른 TTI)"와 "인터랙티브"를 둘 다 달성하되,
RSC / Next.js / react-router 를 쓰지 않는다.

---

## 핵심 설계 원칙

1. **불필요한 것은 만들지 않는다.** 프레임워크의 범용 기능을 재현하지 않는다.
2. **정적인 것은 클라이언트로 JS를 보내지 않는다.** SSR HTML로 그리고 hydrate 대상에서 제외한다.
3. **캐시는 순수 최적화다.** 캐시가 틀리거나 미스해도 정확성이 깨지지 않게 설계한다.
4. **전환은 두 등급으로 나눈다.** 경량 전환만 부분 교체하고, 중량 전환은 브라우저에 위임한다.
5. **생명주기 정리는 React unmount에 위임한다.** 수동 cleanup 시스템을 만들지 않는다.

## 아키텍처 구성요소

### 1. 페이지별 SSR HTML
- 각 페이지를 독립적으로 SSR하여 완성된 HTML을 서빙한다. (MPA 기반)
- 목적: 빠른 첫 화면(FCP), SEO, 페이지별 경량 번들.
- 정적 영역(footer, 헤더, 본문 콘텐츠 등 인터랙션 없는 부분)은 SSR HTML로만 존재하고
  **hydrate하지 않는다.** 해당 영역의 컴포넌트 JS는 클라이언트로 보내지 않는다.
- 렌더링 API: `react-dom/server`의 `renderToReadableStream` (React 19 기준, Bun/웹표준 스트림).

### 2. 이중 React 루트 (Dual Root)
```
#root            ← 바깥 루트. 셸(네비게이션/레이아웃). 한 번 마운트 후 영속.
  └─ #render     ← 안쪽 루트. 페이지 콘텐츠. 전환마다 교체.
```
- **바깥 루트**: 최초 1회 `hydrateRoot(#root, <Shell/>)`. 페이지 전환에도 죽지 않는다.
- **안쪽 루트**:
  - 첫 진입 시: `hydrateRoot(#render, <Page/>)` (SSR HTML 재활용 → 깜빡임 없음)
  - 이후 경량 전환 시: `innerRoot.unmount()` → `createRoot(#render).render(<NewPage/>)`
- **cleanup은 unmount가 자동 처리**한다. (이전 페이지의 effect cleanup / 리스너 / 구독 / 타이머
  정리를 React 언마운트 생명주기에 위임. 수동 teardown 규약을 만들지 않는다.)
- 바깥/안쪽은 **같은 번들·같은 React 인스턴스**를 공유한다. (마이크로 프론트엔드식 분리 빌드 아님)
  → React 중복 인스턴스 없음. "단일 React, 다중 루트" 구조.

### 3. 화면키 캐시 (1회용 핸드오프 버퍼)
- 목적: SSR이 이미 수행한 depth-0 데이터 페치 결과를, 첫 CSR hydration의 보간에 재사용.
  클라이언트가 동일 데이터를 재계산/재페치하지 않게 한다.
- 동작:
  1. SSR 시점에 페이지의 depth-0 API(초기 렌더에 필요한 최상위 호출)를 실행.
  2. 그 결과를 화면키(예: 경로 `/page/1`)로 서버 캐시에 저장하며 HTML을 그린다.
  3. 클라이언트는 hydrate 시 같은 페치 코드로 `/api/page/1`을 요청한다.
  4. 서버는 캐시 히트 시 저장된 값을 즉시 반환(재계산 없음). → 그려진 데이터 == 받은 데이터.
  5. 데이터 일치 + `hydrateRoot` → 마크업 일치 → DOM 재활용 → 깜빡임 없음.
- 성격: **1회용 핸드오프**. 데이터 일관성 보장 장치가 아니라 "SSR 계산을 CSR로 넘기는 통로".
- 수명: SSR ~ 첫 CSR 보간 (수백 ms ~ 수 초).
- **단일 인스턴스 기준**으로 구현. + **TTL** 적용 (보간 전 만료되지 않도록 수명보다 여유 있게,
  예: 수 초 ~ 수십 초. 짧은 쪽 실수가 더 위험하므로 여유를 둔다).
- **다중 인스턴스 환경**: 캐시 미스가 정확성을 깨지 않으므로(미스 시 실제 API를 한 번 더 호출하면 됨)
  확률적 캐싱으로 둔다. 공유 저장소(Redis 등)를 강제하지 않는다. 미스 = 중복 호출 1회 비용일 뿐.

### 4. 데이터 보간 & hydration 규칙
- 첫 진입 hydration은 **반드시 `hydrateRoot`** 사용 (`render`로 교체하지 말 것 → 깜빡임 원인).
- 깜빡임 없음의 전제 = depth-0 확정 데이터의 일치. 화면키 캐시가 이를 공급.
- **비결정적 렌더(시간/랜덤/로케일 등)는 mismatch를 허용한다.** 해당 부분만 깜빡이며 리렌더되는 것은
  의도된 동작이다. 모든 것을 결정적으로 묶으려 하지 말 것.
- 2번째 턴 이후(인터랙션에 의한 갱신)는 캐시 없이 정식 fetch → 최신 데이터 반영. (일반 CSR 동작)

### 5. 페이지 전환 전략 (2등급)
- **경량 전환** (같은 셸 내에서 콘텐츠만 변경):
  - global script가 내부 링크 클릭을 가로챈다 (`e.preventDefault()`).
  - 해당 URL로 fetch → 새 페이지 콘텐츠 획득.
  - 안쪽 루트 `unmount → createRoot().render`로 `#render`만 교체.
  - `history.pushState`로 URL 갱신. `popstate`(뒤로가기) 처리.
- **중량 전환** (레이아웃/컨텍스트가 크게 바뀜):
  - 부분 교체하지 않는다. `location.href = ...`로 **브라우저 네비게이션에 위임.**
  - 정리는 브라우저가 전담(좀비 리스너 0). 부분 교체 복잡도를 짊어지지 않는다.
- 등급 판정 기준은 구현 시 정의 (예: 라우트 메타에 `transition: 'light' | 'heavy'` 표기).

## 명시적으로 만들지 않을 것 (Non-Goals)

다음은 의도적으로 구현하지 않는다. "있으면 좋을까?" 싶어도 추가하지 말 것.

- ❌ **RSC (React Server Components)** — 번들러 통합 비용 회피. 정적부 JS 감축은 페이지 분할 +
  hydrate 제외로 달성. (단, "정적/인터랙티브 혼합도"가 매우 높아지면 재검토 — 아래 미결정 참고)
- ❌ **Next.js** — 불필요한 범용 기능(다층 캐싱, FS 라우팅, 미들웨어 체인, 렌더링 모드 판별 등)이
  response 경로에 오버헤드로 작용.
- ❌ **react-router** — 필요한 라우팅은 위 "경량/중량 전환"의 얇은 global script로 직접 처리.
- ❌ **다층 캐싱 시스템 / 자동 무효화** — 화면키 캐시(1회용)만 둔다.
- ❌ **파일시스템 라우팅 규약** — 명시적 라우트 맵을 코드로 둔다.
- ❌ **이미지/폰트 최적화** — CDN/엣지에 위임. 프레임워크는 자산 URL을 그대로 통과시킨다.
- ❌ **수동 cleanup 생명주기 시스템** — React unmount에 위임.
- ❌ **분리 빌드 마이크로 프론트엔드** — 이중 루트는 같은 번들/단일 React로 한다.

## Bun 빌드 관련 확정 사실 (중요)

- `Bun.build()` JS API 단독으로는 `"use client"` 기준 server/client 그래프 분할을 **하지 않는다.**
  일반 `splitting: true`는 공유 코드/동적 import 기준일 뿐 지시어를 보지 않는다.
- Bun의 RSC 지원은 `--server-components` (+ `--app` / Bun Bake) experimental 경로 전용이며,
  `Bun.build()` 옵션에는 RSC 필드가 없다.
- **본 설계는 RSC를 쓰지 않으므로 위 제약은 문제되지 않는다.** 클라이언트 번들은 페이지 단위
  엔트리포인트로 일반 빌드한다.
- 청크가 하나로 나오는 것은 splitting 미설정/단일 엔트리의 정상 동작이며, 본 설계에서 문제 아님.

## 미결정 (구현 진행하며 확정할 것)

1. **이중 루트 간 상태 공유 방식** — 추후 결정. 같은 React 인스턴스이므로 전역 store / 커스텀 이벤트 등
   필요 시 얹는다. 지금 구조를 제약하지 않음. (보류)
2. **정적/인터랙티브 혼합도** — 한 페이지 안에 작은 인터랙티브 위젯이 정적 콘텐츠 사이에 많이 흩어져 있으면
   "이건 hydrate / 저건 안 함"의 수동 경계 관리 부담이 커진다. 이 부담이 실제로 커지면 그때 RSC식 자동
   경계 분리의 부분 도입을 재검토. (현재는 페이지 성격이 비교적 깔끔하다는 전제)
3. **전환 등급 판정 메커니즘** — light/heavy를 라우트 메타로 표기할지, 휴리스틱으로 판정할지.

## 구현 우선순위 (제안)

1. SSR 렌더 파이프라인 (`renderToReadableStream` + 페이지별 HTML 응답).
2. 화면키 캐시 (단일 인스턴스 + TTL, 미스 안전).
3. 클라이언트 부트: 바깥 셸 `hydrateRoot` + 안쪽 `#render` `hydrateRoot` + 캐시 보간 데이터 fetch.
4. 경량 전환 global script (링크 가로채기 / 안쪽 루트 교체 / pushState / popstate).
5. 중량 전환 위임 + 전환 등급 판정.
6. (그 후) 상태 공유, 최적화.

## 검증 체크리스트

- [ ] 정적 영역(footer 등)의 컴포넌트 JS가 클라이언트 번들에 포함되지 않는다.
- [ ] 첫 진입 시 hydration mismatch 경고가 없다(비결정적 부분 제외).
- [ ] 화면키 캐시 미스 시에도 정상 동작한다(실제 API 재호출).
- [ ] 캐시 TTL이 보간 완료 전 만료되지 않는다.
- [ ] 경량 전환 시 흰 화면 깜빡임이 없다.
- [ ] 경량 전환 반복 시 좀비 리스너/메모리 누수가 없다(unmount 정리 확인).
- [ ] 중량 전환은 브라우저 네비게이션으로 처리된다.
